## Objective

When complete, the HubSpot `crm.record.tab` extension will:

1. Authenticate every API request using HubSpot's signed-request model — the backend verifies the `X-HubSpot-Signature-v3` header (or current equivalent) against the app's client secret, and trusts the `portalId` / `userId` HubSpot auto-appends to the request body/metadata. No bearer-token bypass in production.
2. Fetch snapshots via `hubspot.fetch()` from the extension — HubSpot handles signing and metadata injection; the extension does NOT pass portal/user identity as arbitrary custom headers.
3. Render real reasons-to-contact generated by the tenant's own configured LLM (Anthropic, OpenAI, Gemini, OpenRouter, or a custom OpenAI-compatible endpoint)
4. Render real evidence from live providers (Exa, HubSpot enrichment, news) with dedup + freshness hygiene applied
5. Show an evidence drill-in with per-item provenance, freshness, trust breakdown, and raw provider payload (redacted when restricted)
6. Show one lightweight next-move recommendation per snapshot (hidden for restricted + ineligible)
7. Preserve every Slice 1 guarantee: 8 QA states, zero-leak restricted, tenant isolation, no fabrication of people
8. All tenant API keys encrypted at rest with AES-256-GCM; decryption happens inside the adapter factory only

## Problem Statement

Slice 1's adapters, auth, and encryption are deliberate stubs that let the domain shape stabilize. They are not safe to ship:

- **Base64 encryption** offers no confidentiality. A DB read exposes every tenant's provider keys.
- **Bearer-token middleware** accepts any header matching a portal — no HubSpot verification — so any caller can impersonate any tenant. The Slice 2 middleware verifies HubSpot's signed-request headers instead, which is the transport model HubSpot UI extensions actually use. Refs: [Fetching data for UI extensions](https://developers.hubspot.com/docs/apps/developer-platform/add-features/ui-extensions/fetching-data), [OAuth token introspection](https://developers.hubspot.com/docs/api-reference/latest/authentication/manage-oauth-tokens).
- **Mock LLM + mock signal adapters** return deterministic fixtures, so the product has no real value surfaced to HubSpot users.
- **useSnapshot default fetcher** resolves with local fixtures; the extension is not wired to the API at all — Slice 2 replaces it with `hubspot.fetch()`.

PRD Slice 2 UX layers (richer evidence drill-in, lightweight next-move) were gated on "Slice 1 works." Slice 1 works. They are unblocked.

## Solution Approach

Three implementation phases mapped to dependency layers, executed sequentially in one worktree. TDD is mandatory per CLAUDE.md — every implementation agent writes failing tests FIRST.

- **Phase 0 (Preflight):** confirm base-branch state (Slice 1 merged to `main`, or Slice 2 stacked on `feature/slice-1-core-domain`), worktree spawns cleanly, test portal access works, required env vars are present, and current HubSpot/Drizzle/Hono docs verified via Context7 or the HubSpot docs site. If any fail, STOP and fix.
- **Phase 1 (Security Foundation):** AES-256-GCM + KEK, HubSpot signed-request middleware, independent security audit gate.
- **Phase 2 (Core Infrastructure):** config resolver v2, rate limiter, observability.
- **Phase 3 (Live Adapters):** LLM factory + per-provider adapters, signal adapters, hygiene layers.
- **Phase 4 (Application + UX):** real fetcher, evidence drill-in, next-move, HubSpot test portal seed.
- **Phase 5 (Final Validation):** cross-tenant suite extension, full validation on live test portal.

**Contract discipline:** every Slice 1 interface (`LlmAdapter`, `ProviderAdapter`, `encryption.encrypt/decrypt`, `tenantMiddleware`, `useSnapshot` fetcher, `TrustEvaluator`) is stable. Slice 2 replaces implementations behind those contracts, never reshapes call sites, except where we are intentionally adding new surfaces (evidence drill-in, next-move).

**Security gate before Phase 2:** nothing in Phase 2+ reads a plaintext key or serves a real API call until Phase 1 code is merged to the slice branch **and** the independent `security-audit` review has returned PASS. If the Slice 1 PR lands concurrently, Phase 1 can rebase onto `main`; if not, Phase 1 stays stacked on `feature/slice-1-core-domain`.

**Configurable, never hardcoded:** rate limits, staleness thresholds, allow-lists, provider enablement, model selections — all resolve from `provider_config` / `llm_provider_config` per tenant.

**Auto-commit after each task.** Post-commit hook auto-pushes to `origin/feature/slice-2-live-integrations`.
